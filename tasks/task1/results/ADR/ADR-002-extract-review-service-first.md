# ADR-002: Review Service - первый кандидат для выноса

**Дата:** 2026-02-11
**Автор:** Валерий
**Связан с:** [ADR-001](ADR-001-migration-to-microservices.md)

---

## Контекст

Ресурсы ограничены. Нужно выбрать один модуль для первой экстракции по паттерну Strangler Fig. Выбор должен быть обоснован: минимальный риск, максимальная автономность, четкая граница домена.

---

## Анализ зависимостей

```
BookingService -> UserService -> HotelService -> ReviewService (метод isTrustedHotel) -> PromoCodeService

PromoCodeService -> UserService

ReviewService - нет зависимостей от других сервисов
UserService   - нет зависимостей от других сервисов
HotelService  - нет зависимостей от других сервисов
```

### Оценка кандидатов

| Критерий                              | ReviewService      | UserService         | HotelService       | PromoService              |
| ------------------------------------- | ------------------ | ------------------- | ------------------ | ------------------------- |
| Зависит от других сервисов            | Нет                | Нет                 | Нет                | Да (UserService)          |
| Количество потребителей в монолите    | 1 (BookingService) | 2 (Booking + Promo) | 1 (BookingService) | 1 (BookingService)        |
| Количество затронутых мест при выносе | 1                  | 2                   | 1                  | Невозможно до UserService |
| Изолированность данных (таблица)      | Полная             | Полная              | Полная             | Полная                    |
| Наличие записи из других модулей      | Нет                | Нет                 | Нет                | Нет                       |
| Четкая доменная граница               | Да                 | Да                  | Да                 | Да                        |

---

## Решение: ReviewService

### Обоснование

**ReviewService** оптимальный первый кандидат:

1. **Нулевые входящие зависимости от других сервисов.** ReviewService не вызывает ни один внутренний сервис - только свой репозиторий. Вынос не требует изменений в других частях монолита, кроме одной точки.

2. **Единственная точка интеграции с монолитом.** BookingService обращается к ReviewService только в одном месте: `reviewService.isTrustedHotel(hotelId)` внутри `BookingService.validateHotel()`. Заменить один вызов на gRPC-запрос - минимальная и изолированная доработка.

3. **Изолированная таблица данных.** Таблица `review` не имеет ссылочных ограничений к другим доменным таблицам на уровне бд. Миграция данных - простой `pg_dump` без сложных зависимостей.

4. **Только чтение из других модулей.** Никакой другой модуль не пишет в таблицу `review`. Нет распределенных транзакций.

5. **UserService не подходит первым.** Несмотря на аналогичное отсутствие зависимостей, UserService самый критичный т.к. от него зависят одновременно и BookingService, и PromoCodeService. Вынос требует изменения двух мест и двойного риска регрессий.

6. **HotelService - хорошая альтернатива**, но при выносе возникает вопрос - как Booking Service узнает об изменении `fullyBooked`/`operational`? Нужна Kafka-интеграция, которую удобнее внедрять не на первом шаге.

## План выноса Review Service

### Фаза 0: Подготовка в монолите

- Обернуть прямой вызов `reviewService.isTrustedHotel()` за интерфейс `TrustChecker`
- Реализовать `InternalTrustChecker` (текущая логика) и `RemoteTrustChecker` (gRPC-клиент)
- Переключать реализацию через feature flag

### Фаза 1: Создание Review Service

- Новый Spring Boot проект `review-service`
- Перенести `ReviewService`, `ReviewRepository`, `Review` entity
- Добавить отдельную PostgreSQL-базу `review_db`
- Реализовать:
  - REST `GET /api/reviews/hotel/{hotelId}` - список отзывов
  - REST `GET /api/reviews/hotel/{hotelId}/trusted` - признак доверия
  - gRPC `IsTrustedHotel(hotelId)` - для вызова из монолита

### Фаза 2: Миграция данных

- Остановить запись в `review` таблицу монолита
- Перелить данные: `pg_dump` таблицы `review` -> `review_db`
- Запустить Review Service

### Фаза 3: Проверка

- Включить feature flag: монолит вызывает оба `TrustChecker` (internal + gRPC)
- Сравнивать ответы, логировать расхождения
- Убедиться в идентичности результатов на реальной нагрузке

### Фаза 4: Переключение

- Переключить feature flag на `RemoteTrustChecker` (gRPC)
- Настроить API Gateway: `/api/reviews/**` -> Review Service
- Убедиться в отсутствии ошибок

### Фаза 5: Очистка монолита

- Удалить `ReviewService`, `ReviewRepository`, `Review` entity из монолита
- Удалить таблицу `review` из монолитной бд
- Удалить `InternalTrustChecker`, оставить только `RemoteTrustChecker`
- Убрать feature flag

## Последствия

### Положительные

- Review Service можно масштабировать независимо
- Устанавливается паттерн для последующих шагов миграции

### Риски

- **Задержки:** синхронный gRPC-вызов вместо in-process. Компенсация: circuit breaker с fallback `trusted=false` при недоступности сервиса
- **Консистентность данных при миграции:** dual-write во время перехода на сервис или короткое техническое окно
