# ADR-001: Миграция Hotelio с монолита на микросервисную архитектуру

**Дата:** 2026-02-11
**Автор:** Валерий

---

## Контекст

Hotelio реализован как монолитное Spring Boot приложение. Весь функционал - бронирования, пользователи, отели, промокоды, отзывы - работает в одном процессе и использует одну PostgreSQL-базу данных.

Текущая архитектура описана в [`c4_curent.puml`](../c4_curent.puml).

### Выявленные проблемы

**1. Единая точка отказа.**
Любая ошибка в любом модуле (например, в ReviewService) блокирует создание бронирований.

**2. Невозможность независимого масштабирования.**
При пиковой нагрузке на бронирования нельзя масштабировать только BookingService - масштабируется весь монолит вместе с Review и Promo, которые не испытывают нагрузки.

**3. Плотная связанность.**
BookingService напрямую вызывает 4 других сервиса. PromoCodeService вызывает UserService. Изменение интерфейса одного модуля требует немедленного изменения вызывающего.

**4. Отсутствие DTO-слоя.**
Контроллеры возвращают бд-сущности напрямую (`ResponseEntity<AppUser>`, `ResponseEntity<Hotel>`). Изменение схемы бд ломает API-контракт.

**5. Жестко зашитая бизнес-логика.**
Цены (VIP: `80.0`, обычный: `100.0`) и рейтинги (`rating >= 4.0`, `count >= 10`) захардкожены в коде - не конфигурируются.

**6. Общая база данных.**
Все модули работают с одной бд. Нет изоляции данных между доменами.

**7. Отсутствие пагинации.**
`GET /api/bookings` без `userId` возвращает все записи без ограничений.

---

## Решение

Поэтапный переход на микросервисную архитектуру по паттерну **Strangler Fig**: постепенная замена функциональности монолита независимыми сервисами через API Gateway, без единовременного переписывания.

Целевая архитектура описана в [`c4_target.puml`](../c4_target.puml).

### Принципы целевой архитектуры

- **Database-per-service:** каждый сервис владеет своей бд, никаких shared tables
- **API Gateway** - единая точка входа: маршрутизация, аутентификация, rate limiting
- **gRPC** для синхронных межсервисных вызовов (зависимости gRPC уже добавлены в `build.gradle`)
- **Kafka** для асинхронных событий (уже есть в `docker-compose.yml`)

### Межсервисное взаимодействие

| Сценарий                                  | Паттерн            | Обоснование                                      |
| ----------------------------------------- | ------------------ | ------------------------------------------------ |
| Создание бронирования (проверки)          | gRPC               | Низкая задержка, строгий контракт через Protobuf |
| Обновление статусов (отель, пользователь) | Kafka              | Decoupling, нет блокировки                       |
| Публичный API для клиентов                | REST через Gateway | Удобно для клиентов                              |

## Порядок миграции (Strangler Fig)

Детальный план первого шага - в [ADR-002](ADR-002-extract-review-service-first.md).

```
Шаг 0: API Gateway (проброс на монолит)
Шаг 1: Review Service     - первый (см. ADR-002)
Шаг 2: User Service       - критичный, нужен до Promo
Шаг 3: Hotel Service
Шаг 4: Promo Service      - зависит от User Service
Шаг 5: Booking Service    - последний (максимум зависимостей)
```

На каждом шаге:

1. Поднимаем новый сервис с собственной бд
2. Мигрируем данные
3. Запускаем параллельно, сравниваем ответы
4. Переключаем Gateway и внутренние вызовы монолита
5. Удаляем соответствующий модуль из монолита

## Рассмотренные альтернативы

### Переписать все сразу

**Отклонен.** Система не работает до полного завершения.

### Модульный монолит как промежуточный этап

**Частично принят.** Перед выносом каждого сервиса вводится интерфейс-контракт внутри монолита. Снижается риск регрессий.

---

## Технический долг, который нужно погасить в процессе

1. Добавить DTO-слой (отделить API-контракт от JPA-сущностей)
2. Добавить пагинацию к `GET /api/bookings`
3. Вынести цены и рейтинги в env'ы

## Последствия

### Положительные

- Независимое масштабирование каждого сервиса
- Независимые релизы команд
- Изолированные отказы (circuit breaker паттерн)

### Отрицательные / Риски

- Сетевые задержки вместо in-process вызовов - компенсируется gRPC и кэшированием
- Распределенные транзакции - нет ACID между сервисами, нужен Saga pattern для BookingService
- Операционная сложность - нужны мониторинг, трейсинг, service discovery
- Дублирование ключей (userId, hotelId) без FK-ограничений на уровне бд
