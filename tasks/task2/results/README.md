## Стратегия миграции

### As Is

Монолит `hotelio-monolith` обрабатывает бронирования полностью: хранит данные в собственной PostgreSQL, содержит бизнес-логику (валидация пользователей, отелей, промокодов, расчёт цены).

### Стратегия миграции данных при запуске нового сервиса

1. **Двойная запись не требуется.** Исторические данные из монолита остаются в `monolith-db`. Новые бронирования с момента переключения уходят в `booking-db` (booking-service) и асинхронно реплицируются в `booking_history` через Kafka.
2. **Обратная совместимость.** До момента полного отключения монолита оба источника сосуществуют. Читать полную историю можно через:

- `GET /api/bookings?userId=` — монолит (старые данные)
- `GET /history` — booking-history-service (новые данные через Kafka)

3. **Дата отсечения.** Фиксируется момент включения `BOOKING_SERVICE_EXTERNAL_HOST`. Всё до этого момента — в монолите, всё после — в booking-service.

### To Be

```
Client
  REST -> hotelio-monolith (только read-модели: отели, пользователи, отзывы)
  REST -> booking-service  (create/list bookings)
    Kafka -> booking-history-service (аналитика, аудит)
```

- Монолит полностью лишается модуля бронирований.
- `booking-service` — единственный способ бронирования.
- `booking-history-service` — read-only проекция для аналитических запросов (`SELECT *`).
- Связь между сервисами через Kafka, без прямых бд-зависимостей.
